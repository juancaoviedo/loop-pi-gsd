# Phase 3: Phase Reader and Execution Orchestrator - Specification

**Created:** 2026-07-18
**Ambiguity score:** 0.15 (gate: <= 0.20)
**Requirements:** 4 locked

## Goal

Deliver deterministic orchestration behind `/flow-execute-all-phases` that can load roadmap phases, select target phases safely, prepare isolated execution runs, and package deterministic responder context for interactive workflow rounds.

## Background

Phase 1 established root-level Flow command routing and policy boundaries in `flow/` with a thin gsd-pi bridge.
Phase 2 established bootstrap ingestion and deterministic phase generation.
Phase 3 now introduces execution orchestration entry behavior: identifying what phase(s) should run next, creating run contexts, and preparing deterministic handoff payloads for lifecycle execution and interactive delegation.

This phase does not execute the full lifecycle loop itself (that remains Phase 4). It creates the orchestration substrate needed for safe and reliable execution.

## Requirements

1. **PHASE-01 (Phase selection)**
   - Current: no deterministic roadmap phase reader/selector exists in root Flow.
   - Target: `/flow-execute-all-phases` can read ROADMAP and select phase targets according to explicit selection policy.
   - Acceptance: For a valid roadmap, selected phases are deterministic and reproducible for the same state and selection inputs.

2. **SAFE-02 (Isolation)**
   - Current: no dedicated Flow run isolation manifest exists for selected phase execution contexts.
   - Target: each selected phase run emits an isolated execution context descriptor (run id, workspace scope, artifacts, allowed tools profile).
   - Acceptance: parallel selected phases receive distinct run descriptors with no shared mutable context path.

3. **INTD-01 (Interactive interception routing)**
   - Current: interactive rounds are not represented in Flow orchestration payloads.
   - Target: orchestration layer can declare interception rules for interactive rounds in discuss/spec workflows and route them to responder handling.
   - Acceptance: orchestration output includes explicit interactive interception metadata for downstream execution engine.

4. **INTD-02 (Deterministic context packs)**
   - Current: responder context packs are not generated by Flow orchestration.
   - Target: orchestrator produces deterministic context packs for responder use from project artifacts and selected phase context.
   - Acceptance: same project state and phase input produce stable context-pack payloads.

## Boundaries

**In scope:**
- Root Flow phase reader and selector modules.
- Deterministic selection policy and filtering logic.
- Isolated run descriptor generation for selected phases.
- Deterministic responder context-pack generation and interactive interception metadata.
- Tests for selection determinism, isolation descriptor uniqueness, and context-pack stability.

**Out of scope:**
- Running discuss/plan/execute/verify chain steps directly (Phase 4).
- Retry/repair loops and verification verdict enforcement (Phase 4).
- Long-term evidence governance and escalation policy finalization (Phase 5).

## Constraints

- Flow orchestration code remains in root `flow/`; `external/gsd-pi` remains thin bridge only.
- Selection decisions must be deterministic and idempotent for the same roadmap/state inputs.
- Run descriptors must guarantee isolated context paths per selected phase.
- Context packs must include only allowlisted artifact inputs and stable schema fields.

## Acceptance Criteria

- [ ] PHASE-01: Orchestrator can load roadmap phases and deterministically select target phases.
- [ ] SAFE-02: Orchestrator emits isolated execution descriptors per selected phase run.
- [ ] INTD-01: Interactive interception/routing metadata is emitted for downstream lifecycle execution.
- [ ] INTD-02: Deterministic responder context packs are generated from project artifacts and phase context.
- [ ] Placement rule holds: implementation in root `flow/`, bridge-only seam in `external/gsd-pi`.

## Prohibitions (must-NOT)

- Must not execute full lifecycle chain steps in this phase.
- Must not share mutable run context directories across parallel selected phases.
- Must not generate responder context from non-allowlisted sources.
- Must not embed orchestration logic into deep `external/gsd-pi` internals.
